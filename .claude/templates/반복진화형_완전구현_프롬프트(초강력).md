# 🔁 TEMPLATE OMEGA — 반복 진화형 완전 구현 시스템

> **기획-생성-평가 루프** × **Phase 기반 구현 명령서**
> 두 시스템의 합체. 매 Phase마다 3-에이전트가 돌고, 통과해야만 다음 Phase로 간다.

---

## 이 템플릿이 뭔가요?

기존 두 템플릿의 한계:
- **기획-생성-평가 프롬프트**: 반복 개선은 되지만, 구현 단계가 뭉뚱그려져 있음
- **Template D**: 단계는 정밀하지만, 각 단계 안에서 품질 루프가 없음

**OMEGA는 둘을 결합한다:**
- Template D의 8개 Phase가 **뼈대**
- 매 Phase마다 기획자→생성자→평가자가 **순환**
- 평가자가 통과시켜야만 다음 Phase **게이트가 열림**
- 한 Phase 안에서 최대 3회 반복 후에도 통과 못하면 → **이전 Phase로 롤백**

```
Phase 0 → [기획→생성→평가] → 통과? → Phase 1 → [기획→생성→평가] → 통과? → ...
              ↑       ↓ 반려                           ↑       ↓ 반려
              └───────┘                                └───────┘
              (최대 3회)                                (최대 3회)
                  ↓ 3회 실패                                ↓ 3회 실패
            이전 Phase 롤백                           이전 Phase 롤백
```

## 절대 규칙

1. Phase 순서를 건너뛰지 마세요.
2. 매 Phase에서 [기획자] → [생성자] → [평가자] 순서를 반드시 지키세요.
3. [평가자]가 통과(PASS)시키지 않으면 다음 Phase로 절대 진행하지 마세요.
4. 반려(FAIL) 시, 평가자의 피드백을 [생성자]에게 그대로 전달하여 재작업합니다.
5. 같은 Phase에서 3회 반려 시, [기획자]에게 롤백하여 스펙 자체를 재검토합니다.
6. 같은 Phase에서 기획 재검토 후에도 실패 시, 이전 Phase로 완전 롤백합니다.
7. 각 에이전트 전환 시 반드시 구분선과 역할 헤더를 표시하세요.
8. 추측 금지 — 불확실한 건 진행 전에 반드시 질문하세요.

## PHASE 0 — 완전 탐색
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### [기획자] Phase 0

코드를 한 줄도 짜기 전, 아래를 전부 파악하고 정리합니다.

탐색 범위:
- 프로젝트 전체 폴더/파일 구조
- 관련 기존 컴포넌트/로직 전부
- 기존 디자인 패턴 (컬러, 폰트, 간격, 컴포넌트 스타일)
- 상태관리 방식 (Context / Zustand / useState 등)
- API 호출 패턴
- 타입/인터페이스 구조
- 재사용 가능한 기존 컴포넌트 목록

### [생성자] Phase 0

기획자의 탐색 지시에 따라 실제로 코드베이스를 조사하고, 구조화된 탐색 보고서를 작성합니다.

출력:
```
파악한 기존 패턴: ...
재사용할 컴포넌트: ...
새로 만들어야 할 것: ...
잠재적 충돌/주의사항: ...
디자인 토큰 (컬러, 폰트, 간격): ...
상태관리 현황: ...
```

### [평가자] Phase 0

탐색 보고서를 아래 기준으로 채점합니다 (10점 만점, 7점 이상 통과):

| 기준 | 설명 |
|------|------|
| 완전성 | 관련 파일/패턴을 빠뜨린 게 없는가 |
| 정확성 | 파악한 내용이 실제 코드와 일치하는가 |
| 구조화 | 정보가 다음 Phase에서 바로 쓸 수 있게 정리되었는가 |
| 리스크 식별 | 잠재적 충돌이나 주의사항을 발견했는가 |
| 재사용 판단 | 기존 자원의 재사용 가능 여부를 정확히 판단했는가 |

→ PASS: Phase 1 진행
→ FAIL: 빠뜨린 영역 지적 후 [생성자] 재탐색

- 로딩 상태 정의 (어디서 로딩이 필요하고, 사용자에게 어떻게 보여야 하는지)
